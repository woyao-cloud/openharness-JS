# v2.25.0 "Visibility" — Design Spec

**Date:** 2026-04-30
**Status:** Draft
**Tier:** U-C1 + U-C2 + U-C3 (UI/UX-parity plan, ~/.claude/plans/2-typescript-sdk-moonlit-hinton.md)
**Target release:** v2.25.0 — single PR, ~3 days

## Context

v2.24.0 (shipped 2026-04-30) closed Tier U-B of the 2026-04-27 UI/UX-parity plan, leaving only Tier U-C in the audit. Tier U-C is split into v2.25.0 "Visibility" (C1 + C2 + C3, small renderer polish) and a deferred v2.26+ "Rich + Nested" (C4 + C5, requires UX spec for tables/JSON tree and a query event-protocol RFC for nested tool-call display).

This spec covers **only v2.25.0**. After it ships, OH's interactive REPL surface matches Claude Code's stable surface where parity is meaningful; the remaining U-C4/U-C5 work is intentionally deferred until UX is spec'd.

Two small carry-forward cleanups from PR #91 are folded into this release because they touch the same renderer/repl files: (1) drop the unused `_elapsed` line at `src/repl.ts:1024`, and (2) migrate the surviving `require("../harness/config.js")` at `src/renderer/index.ts:430` (inside `handlePermissionKey`) to a static import (per the v2.23.0 ESM-`require()`-swallow footgun memory).

## Goals

1. **C1 — differentiated spinner state.** The "Thinking…" spinner should communicate which phase the agent is in: LLM call, tool execution (with the tool name), MCP roundtrip, or multi-tool fan-out. Mirrors Claude Code's verb-form spinner labels.
2. **C2 — multi-line input wrap glyph.** Mark every non-last line of a multi-line input (Alt+Enter) with a dim `↵` continuation glyph so the wrap is visible. Mirrors Claude Code's continuation indicator.
3. **C3 — tool-type color coding.** Replace the single `S_YELLOW` for all non-Agent tool calls with a small category-color map (Read=cyan, Mutate=yellow, Exec=magenta, MCP=green). Agent retains its existing cyan-bold `S_AGENT` style via the short-circuit. Makes a tool-call list scannable at a glance.
4. **Cleanups.** Drop `_elapsed` dead code; statically import `appendToolPermission`.

## Non-goals

- **C4 (rich tool-result display: tables / JSON tree / image renderers).** Deferred to v2.26+ — needs a UX spec and a tool-output schema.
- **C5 (nested tool-call display: Agent → child tools).** Deferred to v2.26+ — needs `parentCallId` plumbed through the entire query event stream.
- **Adding a `spinnerStage` field to `LayoutState` or new query events.** The plan suggested it; this spec rejects it as redundant — `state.toolCalls` already carries every signal needed (see Approach).
- **Configurability for the color map.** No theme override for U-C3 in v2.25; punt until users ask.
- **Custom verbs per tool category** (e.g. "Reading…" vs "Editing…"). v2.25 uses the literal `Running <ToolName>` form for clarity; idiomatic verb forms are a v2.26+ polish item if requested.

## Approach

All three features are pure-renderer changes against existing `LayoutState`. **No new query events, no new state fields, no `query/index.ts` touch.**

For C1 specifically, two implementation paths were considered:

| Approach | Wiring | Trade-off |
|---|---|---|
| **(A1) Pure-render derivation** ✅ chosen | Compute label at render time from `state.toolCalls` | No new state; no parallel source of truth; pure function fully unit-testable |
| (A2) Explicit `spinnerStage` field | Add field to `LayoutState`, emit from `query/index.ts` | Plan's original proposal — extra event channel that has to stay consistent with `toolCalls` |
| (A3) Hybrid | Derive in renderer, expose via state method | Same outcome as A1 with extra ceremony |

A1 wins because `state.toolCalls: Map<string, ToolCall>` already carries `status` and `toolName` for every active call. Deriving the spinner label from it is total, deterministic, and side-effect-free.

## Component design

### Component 1 — Spinner label derivation (C1)

**New file:** `src/renderer/spinner-label.ts`

```ts
export function deriveSpinnerLabel(toolCalls: Map<string, ToolCall>): string {
  const running = [...toolCalls.values()].filter((tc) => tc.status === "running");
  if (running.length === 0) return "Thinking";
  if (running.length === 1) {
    const name = running[0]!.toolName;
    if (name.startsWith("mcp__")) {
      // mcp__<server>__<tool> → "Calling <server>:<tool>"
      const rest = name.slice("mcp__".length);
      const idx = rest.indexOf("__");
      if (idx > 0) return `Calling ${rest.slice(0, idx)}:${rest.slice(idx + 2)}`;
      return `Running ${name}`;
    }
    return `Running ${name}`;
  }
  return `Running ${running.length} tools`;
}
```

**Wiring:** `src/renderer/layout-sections.ts:106-128` — `renderSpinnerSection`. Replace the hardcoded `const thinkText = "Thinking";` with `const thinkText = deriveSpinnerLabel(state.toolCalls);`. The shimmer loop iterates `thinkText.length` so a longer label like `Running NotebookEdit` shimmers across the full label. Suffix (` Ns | <tokens>...`) and elapsed-driven color staging are unchanged.

**Edge cases:**
- `streamingText` or `thinkingText` set → existing line 107 short-circuit applies; spinner section returns early. No change.
- Tool call missing `status` → not possible (status is required in `ToolCall` type).
- Non-`mcp__`-prefixed call with `__` in the name → `running[0]!.toolName` matched as-is; falls through to `Running <toolName>`.

### Component 2 — Wrap glyph in input section (C2)

**Wiring:** `src/renderer/layout-sections.ts:467-498` — `renderInputSection`.

For each `inputLines[li]` where `li < inputLines.length - 1`:
- Compute the column after the line's last character: `glyphCol = inputStart + inputLines[li].length` for the first line, `inputLines[li].length` for subsequent lines.
- Draw `↵` at `(inputRow + li, glyphCol)` with `S_DIM`.
- If `glyphCol >= grid.width`, omit the glyph (line filled the row; can't render past the edge).

**`[N lines]` suffix collision:** when `inputLines.length > 1`, the existing `[N lines]` suffix lands on row 0. If the `↵` for line 0 would land at the same column as `[N lines]`, the suffix wins (it's more informative, and in practice the user sees the line count rather than a continuation indicator on a line that's clearly continued). Implementation: compute the suffix column first, only draw `↵` for `li=0` if `glyphCol < lineCountCol - 1`.

**Last line stays unmarked.** A single-line input (no `\n`) gets no glyph.

### Component 3 — Tool color map (C3)

**New file:** `src/renderer/tool-color.ts`

```ts
const READ = new Set(["Read", "Glob", "Grep", "WebFetch", "WebSearch", "ExaSearch"]);
const MUTATE = new Set(["Edit", "Write", "NotebookEdit"]);
const EXEC = new Set(["Bash", "PowerShell"]);

export function toolColor(toolName: string): "cyan" | "yellow" | "magenta" | "green" {
  if (READ.has(toolName)) return "cyan";
  if (MUTATE.has(toolName)) return "yellow";
  if (EXEC.has(toolName)) return "magenta";
  if (toolName.startsWith("mcp__")) return "green";
  return "yellow"; // fallback preserves prior behavior
}
```

Agent / ParallelAgents are not in this map: the existing `isAgent` branch in `renderToolCallsSection` short-circuits to the existing `S_AGENT` style (cyan-bold, defined at `src/renderer/layout-sections.ts:22`) before `toolColor()` is consulted.

**Wiring:** `src/renderer/layout-sections.ts:161-162` — `renderToolCallsSection`.

Replace:
```ts
const statusStyle = tc.status === "error" ? S_ERROR : tc.status === "done" ? S_GREEN : isAgent ? S_AGENT : S_YELLOW;
const nameStyle = isAgent ? S_AGENT : { ...S_YELLOW, bold: true };
```

with:
```ts
const toolStyle: Style = { fg: toolColor(tc.toolName), bg: null, bold: false, dim: false, underline: false };
const statusStyle = tc.status === "error" ? S_ERROR : tc.status === "done" ? S_GREEN : isAgent ? S_AGENT : toolStyle;
const nameStyle = isAgent ? S_AGENT : { ...toolStyle, bold: true };
```

`S_YELLOW` retention: `S_YELLOW` is also used at `layout-sections.ts:461` (`renderHelpSection` ⚡ note) — keep it in place there; we're not touching unrelated yellow uses.

`S_AGENT`: defined at `src/renderer/layout-sections.ts:22` as `s("cyan", true)` (cyan-bold). The `isAgent` branch in `renderToolCallsSection` short-circuits before `toolColor()` is consulted, so Agent calls keep their existing cyan-bold rendering. Agent is intentionally not in `toolColor`'s switch.

### Component 4 — Cleanups

- **`src/repl.ts:1024`:** delete the line `const _elapsed = prevTc?.startedAt ? Math.floor((Date.now() - prevTc.startedAt) / 1000) : 0;`. Computed but never read.
- **`src/repl.ts` `handlePermissionKey`:** replace `const { appendToolPermission } = require("../harness/config");` (or the equivalent dynamic call site) with a static `import { appendToolPermission } from "./harness/config.js";` at the top of the file. Verify no circular-import regression by running `npm run build` after the change.

## Data flow

```
LayoutState (already populated by REPL)
├── toolCalls: Map<callId, ToolCall>
│     └── deriveSpinnerLabel() → renderSpinnerSection (C1)
│     └── toolColor(tc.toolName) → renderToolCallsSection (C3)
└── inputText: string
      └── split("\n") → renderInputSection wraps each non-last line with "↵" (C2)
```

No new event flow, no new state, no new harness/query touches.

## Error handling

- `deriveSpinnerLabel`: pure, total. Empty map → `"Thinking"`. Malformed `mcp__` name (missing second `__`) → falls through to `Running ${toolName}`. Cannot throw.
- `toolColor`: pure, total. Unknown tool → `yellow`. Cannot throw.
- Wrap glyph: clipped to `grid.width`; never overflows. Glyph col past edge → omitted (silent).

## Release pattern

Standard from v2.21 / v2.22 / v2.23 / v2.24:
1. Merge the visibility PR(s).
2. Bump `package.json` to `2.25.0`. SDK unchanged this release.
3. Consolidate `Unreleased` → `## 2.25.0 (YYYY-MM-DD) — Visibility`. **Cross-check Unreleased against `git log v2.24.0..HEAD`** before tagging — the rebase-conflict-drops-Unreleased-entries pattern has hit twice in the last six releases.
4. Tag `v2.25.0`, push tag, npm publish via `publish.yml`.
5. Write `project_v2_25_0.md` memory entry; update `MEMORY.md` index.

## Rollback

Pure renderer change with no config schema or state-shape changes. Reverting the PR cleanly restores prior behavior. No data migration; no user-visible config to deprecate.
